RHEL Storage I/O Performance and File System Optimization Study Guide

1. Introduction to Storage Performance Monitoring

Diagnosing storage performance is a critical skill for any systems architect. While performance issues often manifest at the application level, the root cause frequently lies in I/O bottlenecks that create ripples across the entire system.

* Primary Bottleneck Indicator: Using the top command, an architect identifies a storage-constrained system by observing low CPU usage (%us) in conjunction with high CPU wait time (%wa). This delta indicates the CPU is idle, not because of a lack of work, but because it is stalled waiting for the storage subsystem to fulfill data requests. Because Linux load averages include processes in uninterruptible sleep (waiting for disk), this bottleneck will also drive up the system load average despite the low processor activity.

Diagnostic Tool Categories

Category	Purpose	Primary Tools
Device-level Usage	Reports CPU and I/O statistics for physical disks and partitions.	iostat
Process-level Usage	Correlates I/O throughput with specific processes or threads.	iotop
Block-layer Usage	Traces request stages (Queue, Dispatch, Complete) within the kernel.	blktrace, blkparse, btt
Performance Co-Pilot (PCP)	Scalable framework for distributed live and historical analysis.	pcp atop, pmiostat

2. Analyzing Device and Process-Level I/O

Device-Level Metrics with iostat

Install the sysstat package to access this utility: dnf install sysstat

* Command Usage: iostat -xd generates extended statistics for all devices.
* Architect's Note: It is vital to remember that the first report generated by iostat displays the mean statistics since the last system boot. Always look at subsequent reports for current, real-time activity.
* The %util Column: This indicates the percentage of time the device was busy. Values consistently at or above 100% are definitive evidence of a device-level bottleneck.

Process-Level Metrics with iotop

The iotop-c package provides the C-based implementation of this tool: dnf install iotop-c

* Standard Options:
  * -o (--only): Filters out idle processes to focus only on active I/O.
  * -P (--processes): Aggregates thread statistics. This is essential for monitoring multithreaded applications like databases, where I/O might otherwise be fragmented across dozens of threads.
  * -a (--accumulated): Displays total I/O volume since the utility was launched.
* Visual Indicators: The vertical dotted lines (⡇) provide a graphical sparkline of activity, allowing for rapid visual identification of I/O spikes without parsing raw throughput numbers.

Advanced Analysis with Performance Co-Pilot (PCP)

Enable the Performance Metrics Collector Daemon (pmcd) to begin: systemctl enable --now pmcd

* Tool Comparison:
  * pcp atop -d (Process-Centric): Best for identifying which specific PID is driving system changes. It distinguishes between data sent to the driver and actual hardware I/O, exposing the impact of file-system caching.
  * pmiostat (Device-Centric): Provides live, high-resolution statistics similar to iostat. Use the -R flag (e.g., pmiostat -R sda) to use regular expressions for targeted device monitoring.

3. Block-Layer I/O Tracing and Interpretation

The Tracing Workflow

Block-layer analysis involves a three-step pipeline:

1. Capture: blktrace -d /dev/sda logs binary events per CPU core.
2. Parse: blkparse -i sda_trace converts binary data into a human-readable timeline.
3. Summarize: btt (for stats) or blkiomon (for histograms) aggregates the raw trace into actionable data.

blkparse Output Field Reference

Field	Description
Device	Major and minor device numbers.
CPU	Core handling the specific I/O event.
Sequence	Chronological counter per CPU.
Timestamp	Elapsed time in seconds since the trace start.
PID	Process ID that triggered the request.
Action	Q (Queued), D (Dispatched to hardware), C (Completed).
RWBS	R (Read), W (Write), B (Barrier), S (Synchronous), D (Discard).

Latency Analysis

* Hardware Latency (D2C): A large gap between "Dispatched" and "Completed" indicates the physical disk or storage array is struggling to service requests.
* Scheduler Latency (Q2D): A large gap between "Queued" and "Dispatched" indicates the kernel's I/O scheduler is backed up, often due to high load or sub-optimal scheduler selection.

4. Multiqueue I/O Schedulers: Selection and Configuration

The blk-mq Mechanism

The Multi-Queue Block I/O Queuing Mechanism scales performance for multi-core systems by mapping I/O to multiple hardware queues. This reduces lock contention when multiple cores perform I/O to a single device, which is essential for NVMe protocols.

Scheduler Comparison

* none: A FIFO algorithm with minimal overhead. Required for NVMe devices. Red Hat recommends not changing the scheduler for NVMe.
* mq-deadline: Guaranteed latency by prioritizing reads and batching LBA-proximate requests. Recommended for virtual guests.
* kyber: Self-tuning based on target latency. Recommended for fast SSDs.
* bfq: Budget-based queuing for fairness. Recommended for HDDs and interactive desktop workloads.

5. Tuning Solid-State Drives (SSDs)

SSD Tuning Strategy

Optimization focuses on reducing write amplification to ensure cell longevity and maximizing speed by bypassing legacy rotational logic.

Access Patterns

* Caches: Configure as write-through; legacy read-ahead/write-behind is unnecessary.
* Journaling: Red Hat discourages journaling on SSDs, as it causes wear through double-writing and reduces overall throughput.

Discard (TRIM) Operations

Method	Mechanism	Recommended Usage
Batch Discard	fstrim [mount-point]	Manual/On-demand; impactful during execution.
Periodic Batch	fstrim.timer	Red Hat Recommended. Runs weekly by default to balance wear and performance.
Online Discard	mount -o discard	Real-time. Can cause performance degradation during heavy file deletions.

6. Optimizing XFS and Ext4 File Systems

File System Selection

Scenario	Recommended File System
Large files, enterprise workloads, multithreaded I/O	XFS (Default)
Predominantly small files, single-threaded I/O, < 1000 IOPS	ext4
Necessity for offline shrinking	ext4

Formatting and Features

* XFS: Adjust inode size (-i size=512) for extended attributes and directory block size (-n size=8192) for directory-heavy workloads.
* Ext4: Use tune2fs -O to toggle features.
  * large_dir: Increases file limits per directory.
  * huge_file: Disable this if small files are predominant to optimize space.
  * Caveat: The extra_isize feature (enabled by default) has no effect and should be disabled if the inode size is set below 256 bytes.

Mount Option Tuning

* Access Times: Use noatime to eliminate metadata writes for file reads.
* XFS Specific: Use logbsize=256k. This is highly effective for systems lacking a nonvolatile write cache, as it handles more operations in memory before a log flush.
* Ext4 Specific: Use journal_ioprio (0-7) to prioritize the journal (0 is highest).

7. Benchmarking with the Flexible I/O Tester (fio)

Essential fio Parameters

Parameter	Description
--name	Job identifier.
--filename	Target file or block device.
--direct=1	Bypasses page cache to measure actual disk performance.
--rw	Pattern (e.g., randrw for mixed workloads).
--bs	Block size (e.g., 4k).
--ioengine	Engine (e.g., libaio for asynchronous I/O).
--iodepth	Number of parallel requests in flight.

Case Study: Database Workload Simulation

Simulating a typical database profile (70% Read / 30% Write) on an XFS partition demonstrates the measurable impact of targeted tuning.

Scenario: 4 Parallel threads, 4G file size, libaio, iodepth=32.

Metric	Before Tuning (Default)	After Tuning (noatime, logbsize=256k)
Read IOPS	21.7k	24.2k
Read Bandwidth	84.7 MiB/s	94.7 MiB/s
Avg. Read Latency (clat)	3563.39 µs	3182.92 µs
Disk Utilization (%util)	79.83%	77.10%

Conclusion: By increasing the log buffer and disabling access time updates, throughput increased by ~11% while simultaneously reducing hardware utilization and latency. Architecturally, benchmarks should always be repeated multiple times to ensure reproducibility and establish a reliable performance baseline.
